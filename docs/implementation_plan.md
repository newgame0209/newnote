## 開発フェーズおよびタスク指針

### フェーズ1: 環境セットアップ

1.  手順1: プロジェクトディレクトリに `/frontend` と `/backend` フォルダを作成します。 (参考: PRD セクション1, 5; プロジェクト概要)
2.  手順2: プロジェクトのルートにGitリポジトリを初期化し、`main` と `dev` ブランチを設定し、GitHubでブランチ保護ルールを構成します。 (参考: PRD セクション1.4)
3.  手順3: Windsurf（近代的なIDE）を用いてプロジェクトの設定を行い、JavaScriptとPythonの両方の開発に対応できるようにします。 (参考: 技術スタック; その他のツール)
4.  手順4: `/frontend` フォルダでNode.jsプロジェクトを初期化し、React, Tailwind CSS, shadcn/ui, Fabric.jsをインストールします。Node.jsのバージョンは環境に応じたものを使用してください。 (参考: 技術スタック: フロントエンド)
5.  手順5: `/backend` フォルダにPython仮想環境を設定し、API開発およびTTSサービス（OpenAI/Google）の統合用に必要なパッケージ（例: FlaskまたはFastAPI, requests）をインストールします。 (参考: PRD セクション5, 技術スタック: バックエンド)
6.  手順6: **検証:** `/frontend` では `npm -v` を、`/backend` では `python --version` を実行して、環境が適切にセットアップされていることを確認します。

### フェーズ2: フロントエンド開発

1.  手順7: `/frontend/src/components/Home.js` にホームページ用のコンポーネントを作成します。このコンポーネントは、日付、カテゴリ、タイプでのノートリストを表示し、グリッドとリストビューの切り替えを可能にします。(参考: PRD セクション3, ユースケース1)

2.  手順8: `/frontend/src/components/NewNoteModal.js` で新規ノート設定モーダルを開発し、用紙サイズ(A4, A3, A7)、向き(縦/横)、色(白/黄色)、メインおよびサブカテゴリ、タイトル入力を行います。送信後はノート編集ページに遷移します。(参考: PRD セクション4, ユースケース1)

3.  手順9: `/frontend/src/components/NoteEditor.js` にノート編集画面コンポーネントを作成します。Fabric.jsキャンバスを統合し、手描き入力やペン、消しゴム、マーカー、テキスト入力、画像アップロードをサポートします。

    *   10ページまでで、スワイプナビゲーションが可能(ページ番号を一時的に表示)。
    *   モバイル/タブレットモードでピンチズームとスクロールを無効にする。
    *   Apple Pencilと指のタップを入力として検出。 (参考: PRD セクション4, ユースケース2)
    
4.  手順10: `/frontend/src/components/NoteEditor.js` でテキスト入力選択時にソフトキーボードが自動表示されるモバイルビュー向けに実装します。(参考: PRD セクション4, ユースケース2)

5.  手順11: `/frontend/src/components/Settings.js` に設定画面を開発し、ユーザーが音声のタイプ(アダルト女性/男性)を切り替えることができるようにします。(参考: PRD セクション4, ユースケース3)

6.  手順12: **検証:** フロントエンドアプリケーションをローカルで実行して（例: `npm start`）、コンポーネントの動作を確認します。

### フェーズ3: バックエンド開発

1.  手順13: 軽量フレームワーク（FlaskまたはFastAPI)を使用してメインのバックエンドアプリファイル `/backend/app.py` を作成します。(参考: PRD セクション5, 技術スタック: バックエンド)

2.  手順14: POST APIエンドポイント `/api/tts` を `/backend/routes/tts.py` に開発し、ノートコンテンツ（手描き文字とテキスト入力）を受け取り、外部TTS API (OpenAIまたはGoogle TTS) を呼び出すことでAI TTSプロセスをトリガーします。(参考: PRD セクション2, AI音声変換)

3.  手順15: ノートのメタデータ（選択された用紙サイズ、向き、色、カテゴリ、タイトル、最後に編集されたページ）を受け取り、SQLiteなどの簡単な永続化方法を利用して保存する簡単なノート保存エンドポイント `/api/notes` を作成します。(参考: PRD セクション2, ノート作成/編集)

4.  手順16: バックエンドでCORSを設定し（例: Flask-CORSを使用）、フロントエンドオリジンからのリクエストを許可します（例: [http://localhost:3000）。(参考](http://localhost:3000%EF%BC%89%E3%80%82\(%E5%8F%82%E8%80%83): 統合, 技術スタック: バックエンド)

5.  手順17: 複数のTTSおよびOCR API呼び出しの遅延をキャプチャするために、APIエンドポイントに堅牢なエラーハンドリングとロギングを追加します。(参考: PRD セクション8, 既知の問題)

6.  手順18: **検証:** `curl` または Postman を使ってエンドポイントをテストします。例として、`curl -X POST http://localhost:5000/api/tts -d '{"text": "テスト文章"}'` を実行し、有効な音声ストリームレスポンスと200レスポンスを確認します。

### フェーズ4: 統合

1.  手順19: ノートエディターコンポーネント(`/frontend/src/components/NoteEditor.js`) で、ユーザーが音声変換ボタンを押すと、現在のノートのテキストコンテンツがバックエンド `/api/tts` エンドポイントに送信されるAPI呼び出しを統合します。(参考: ユースケース2, Q&A: AI音声変換)
2.  手順20: ホーム画面コンポーネントと新規ノートモーダルに、API呼び出しを統合して、新しいノートを作成し、再度開いた際に最後に編集されたページを復元します。(参考: PRD セクション2, ユースケース1 & 2)
3.  手順21: すべてのAPIリクエストに適切なエラーハンドリングと、複数の連続したAPI呼び出しによる遅延が発生した場合のユーザー通知を含めることを確認します。(参考: PRD セクション8, 既知の問題)
4.  手順22: **検証:** ブラウザを使用してノートの作成、編集、保存、音声変換をシミュレートします。オーディオの再生が機能することと、保存されたノートデータが最後に編集されたページを正しく復元することを確認します。

### フェーズ5: デプロイメント

1.  手順23: Reactビルドプロセスを使用してフロントエンドのプロダクションビルドを作成し（例: `npm run build`）、静的アセットのデプロイメントを準備します。(参考: PRD セクション7, ウェブベースのデプロイ)
2.  手順24: PythonバックエンドをDockerを使用してコンテナ化します。`/backend` に `Dockerfile` を作成します(もしDockerをデプロイメントに選んだ場合)。(参考: 技術スタック: バックエンド)
3.  手順25: デプロイメントスクリプトとCI/CDパイプライン（GitHub Actionsまたは別のCIツールを使用）を設定してフロントエンドをS3バケットや類似の静的ホスティングソリューションにデプロイし、バックエンドをクラウドサービス（例: AWS EC2や類似のウェブサービス）にデプロイします。(参考: PRD セクション7, デプロイメント要件)
4.  手順26: **検証:** デプロイされたら、プロダクションURLに対してエンドツーエンドテスト（Cypressを使用するか、手動テストを含める）を実行して、機能性とパフォーマンスを確認します。

### フェーズ6: ポストローンチ

1.  手順27: サーバーとAPIのパフォーマンスを監視し、手描きのOCRに続くTTS変換の逐次的なAPI呼び出しが許容される遅延時間内にあることを確認します。ロギングとエラーアラート設定を行います。(参考: PRD セクション8, パフォーマンスの懸念)
2.  手順28: 初期の少数のユーザーグループ（Q&Aで説明されているように）からユーザーフィードバックを集め、改善点やバグ修正を特定し、将来のモバイル（iOS/Android）への適応を計画します。(参考: Q&A: ユーザーフィードバック)
3.  手順29: コード構造とAPIエンドポイントを詳細にドキュメント化し、PDFアップロードやネイティブモバイルアプリ開発など、将来のイテレーションをシームレスに行えるようにします。(参考: PRD セクション7, 将来の展開)
4.  手順30: **検証:** パフォーマンスメトリックとユーザーフィードバックログを定期的に確認し、エラー率や応答時間が期待される閾値を超えた場合はソリューションを繰り返し改善します。

以上が、ウェブベースのノートアプリ「しゃべるノート」のMVPを開発するためのステップバイステップの計画です。この計画には、ノートの作成、編集、およびAIを活用した音声変換の機能を備え、将来的にモバイルプラットフォームへの拡張を可能にする基盤を構築します。
